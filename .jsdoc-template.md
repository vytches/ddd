# JSDoc Template & Style Guide

## VytchesDDD JSDoc Standards

### LLM-Optimized JSDoc Structure

All public APIs MUST follow this exact template structure for consistency and LLM consumption:

## Class Template

```typescript
/**
 * @llm-summary Brief one-line description of the class purpose
 * @llm-domain [Core|Pattern|Architecture|Integration|Infrastructure] 
 * @llm-complexity [Simple|Medium|Complex|Expert]
 * 
 * @description
 * Detailed description of the class, its responsibilities, and how it fits into DDD patterns.
 * 
 * @example
 * ```typescript
 * // Basic usage example
 * const instance = new ClassName(params);
 * const result = instance.method();
 * ```
 * 
 * @example
 * ```typescript
 * // Advanced usage with error handling
 * const [error, result] = safeRun(() => new ClassName(params));
 * if (error) {
 *   console.error('Creation failed:', error.message);
 * }
 * ```
 * 
 * @since 1.0.0
 * @public
 */
export class ClassName {
  /**
   * @llm-summary Brief description of constructor purpose
   * 
   * @description
   * Detailed description of what the constructor does and any important initialization logic.
   * 
   * @param {ParamType} param1 - Description of parameter 1
   * @param {ParamType} param2 - Description of parameter 2
   * 
   * @throws {ErrorType} When parameter validation fails
   * @throws {ErrorType} When business rules are violated
   * 
   * @example
   * ```typescript
   * const instance = new ClassName('value1', { option: true });
   * ```
   */
  constructor(param1: ParamType, param2: ParamType) {}

  /**
   * @llm-summary Brief description of method purpose
   * @llm-sideeffects [None|Mutates|Async|External]
   * 
   * @description
   * Detailed description of method behavior, business rules, and any side effects.
   * 
   * @param {ParamType} param - Description of parameter
   * @returns {ReturnType} Description of return value
   * 
   * @throws {ErrorType} When validation fails
   * 
   * @example
   * ```typescript
   * const result = instance.methodName(param);
   * ```
   */
  methodName(param: ParamType): ReturnType {}
}
```

## Interface Template

```typescript
/**
 * @llm-summary Brief one-line description of the interface purpose
 * @llm-domain [Core|Pattern|Architecture|Integration|Infrastructure]
 * @llm-contract [Required|Optional|Extends:BaseInterface]
 * 
 * @description
 * Detailed description of the interface, its role in the system, and implementation requirements.
 * 
 * @example
 * ```typescript
 * // Implementation example
 * class ConcreteClass implements InterfaceName {
 *   method(): ReturnType {
 *     // Implementation
 *   }
 * }
 * ```
 * 
 * @since 1.0.0
 * @public
 */
export interface InterfaceName {
  /**
   * @llm-summary Brief description of method purpose
   * 
   * @description
   * Detailed description of method requirements and expected behavior.
   * 
   * @param {ParamType} param - Description of parameter
   * @returns {ReturnType} Description of return value
   * 
   * @throws {ErrorType} When implementation requirements are not met
   */
  method(param: ParamType): ReturnType;
}
```

 *

 * ```
 * 
 * @since 1.0.0
 * @public
 */
export function functionName(param1: ParamType, param2: ParamType): ReturnType {}
```

## LLM-Specific Tags

### Custom JSDoc Tags for LLM Optimization:

- **@llm-summary** - Single line description for quick LLM understanding
- **@llm-domain** - Domain classification (Core, Pattern, etc.)
- **@llm-complexity** - Complexity level (Simple, Medium, Complex, Expert)
- **@llm-contract** - Interface contract type
- **@llm-sideeffects** - Side effects classification
- **@llm-pure** - Pure function indicator
- **@llm-usage** - Usage frequency indicator

### Standard JSDoc Tags (Required):

- **@description** - Detailed description
- **@param** - Parameter documentation
- **@returns** - Return value documentation
- **@throws** - Exception documentation
- **@example** - Code examples (minimum 2 per public API)
- **@since** - Version introduced
- **@public/@internal** - API visibility

## Quality Requirements

### Required Elements:
1. **Every public API** must have complete JSDoc
2. **Minimum 2 examples** per public method/function
3. **Error handling examples** using safeRun pattern
4. **Business context** in descriptions
5. **DDD pattern explanation** where applicable

### LLM Optimization:
1. **Structured information** using custom tags
2. **Consistent terminology** across all documentation
3. **Domain classification** for better context
4. **Complexity indicators** for usage guidance
5. **Usage frequency** for prioritization

## Validation Rules

### ESLint Rules (to be implemented):
- Require JSDoc on all exported items
- Enforce @llm-summary tag
- Require @example tag
- Validate @param/@returns completeness
- Check for @throws documentation

### Quality Gates:
- JSDoc coverage > 95% for public APIs
- All examples must be valid TypeScript
- All @throws must reference actual error types
- Domain classification must be valid
